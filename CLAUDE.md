# ミッション

あなたの目的は2つです。
1. **実装すること**
2. **Docsを作り上げること**

この2つは常に並走する。実装が進んだらDocsを更新し、Docsで方針を固めてから実装する。

# Docsの構成

ビジネス要件・仕様・制約は `docs/` 配下にすべて詰まっている。
- `docs/biz/` … ビジネス要件（コンセプト、市場、ユースケース等）
- `docs/tech/` … 技術要件・データモデル・技術メモ
- `docs/devlog/` … 開発ログ
- `docs/overview.md` … 全体像
起動時にこれらを読み込むこと。ログの記録先・記録方法は既存Docsの運用ルールに従う。

# プラットフォームと技術選定

- **プラットフォームは iOS アプリで固定**とする。ここは変更しない。
- iOS 以外の技術選定（言語・フレームワーク・アーキテクチャ・ライブラリ・
  テスト/静的解析ツール等）は、Docsの要件に基づき **あなたが自律的に選定してよい**。
- 選定した技術と、その選定理由・却下した選択肢を必ずログに残す。

# 参考モック

- `docs/reference/datamine_prototype.jsx` にモックがある。
- これは **あくまで参考**。仕様の正解ではなく、ビジネス要件（Docs）が常に優先される。
- 使えるアイデア（UI構成、画面遷移、データの見せ方など）は取り入れてよいが、
  忠実に再現する義務はない。iOSアプリとして自然な形に翻案してよい。
- モックと Docs が食い違う場合は Docs を優先し、その判断をログに残す。

# 実装の進め方（このループを回す）

**探索 → 計画 → 実装 → 検証 → レビュー → 記録 → 次のタスク**

- 大きなタスクは着手前に小タスクへ分解し、計画をログに書く。
- 実装後は必ず検証する。失敗したら直してから次へ進む。
- 1つのまとまりが完了するごとに、実装内容・判断・検証結果をログに残す。
- **一つのタスクが完了したら、確認を求めずに次のタスクへ自動で進む。**

# 検証（各まとまりの完了前に必ず通す）

- 選定した技術スタックにおける **ビルド・テスト・静的解析** をすべて通すこと。
  失敗したら直してから次へ進む。
- 使用する具体的な検証コマンドは、技術選定の結果として確定させ、ログに明記すること
  （以降のセッションでも参照できるようにする）。

# サブエージェントの活用方針

実装の主導権はメインエージェント（あなた）が持つ。以下はサブエージェントへ隔離する。
- 重い調査（リポジトリ探索、ドキュメント調査、ログ/テスト出力の精査）
  → 中間出力はサブエージェント側に留め、要約だけを受け取る。
- 独立した調査は、依存関係がないものに限り並列で走らせる（順序依存があるものは並列化しない）。
- 第三者レビューは reviewer サブエージェントに委ねる（下記レビューゲート参照）。
固定エージェントを増やしすぎない。隔離したい明確な理由があるものだけに絞る。

# モデルの使い分け

作業の性質に応じてモデルを使い分けること。
- 最難所の設計判断・アーキテクチャ・要件解釈・レビュー：最上位モデル（Opus系）
- 主力の実装作業：高速な上位モデル（Sonnet系）
- 機械的・定型的な作業（一括置換、命名整理、雛形生成）：軽量モデル（Haiku系）
どのモデル/エージェントに何を割り当てたかも、その理由とともにログに残す。

# 自律的判断のルール

実装判断は、ビジネス要件・Docs・これまでの会話から適切に行い、自律的に実装する。

**自分で決めて進めてよいこと**
- 命名、内部設計、ファイル配置、リファクタリング、実装の具体的手段。
- iOS以外の技術選定。
- Docsのファイル・ディレクトリ構成は自由に最適化してよい。既存のファイル構成も変更してよい。

**止まって確認すべきこと（これ以外は止まらない）**
- 要件が本質的に矛盾/欠落していて、推測すると成果物の方向性が変わる場合。
- 破壊的で復旧困難な操作（データ削除、外部への公開・送信、認証情報の扱いなど）。

上記2つ以外では確認を求めず、自律的に判断して進む。
判断に迷った箇所は「なぜその選択をしたか」を必ずログに残し、後から検証可能にすること。

# レビューゲート（第三者レビュー）

実装のまとまりが完了に達したら、reviewer サブエージェントを呼び出しレビューを受ける。
- reviewer は実装過程に関与しない独立した立場で、成果物だけを白紙から評価する。
- reviewer には読み取りと検証のみを許可し、コードは書き換えさせない（指摘に徹する）。
- 🔴（Blocker）が残っている間は「完了」と呼ばない。指摘を潰してから次へ進む。
- レビュー指摘と、それへの対応（修正 or 却下の理由）もログに残す。

# Docs / ログの運用

- Docsは私たちが確認するためであり、同時にあなたが参照するためのもの。常に実装と一致させる。
- 実装ログに限らず、思考フロー・検討した選択肢・却下した案・判断根拠も、細かくすべてログに残す。
  （記録先・記録方法は既存Docsの運用ルールに従う）
- Docsの構成は自由に最適化してよい。人間とあなた自身の双方が辿りやすい構造を優先する。

# 完了条件（Definition of Done）

タスクを「完了」と呼ぶ前に、以下を満たすこと。
- 対象機能が要件どおり動作する。
- 検証（ビルド・テスト・静的解析）がすべて通る。
- reviewer のレビューを受け、🔴（Blocker）が残っていない。
- Docsが実装と一致し、ログに判断と結果が残っている。
