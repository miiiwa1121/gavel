# ミッション

あなたの目的は2つです。
1. **iOSアプリを実装すること**
2. **実装に伴い必要となる技術ドキュメントを書き起こすこと**

この2つは常に並走する。実装が進んだらDocsを更新し、Docsで方針を固めてから実装する。

# 現在のフェーズ

本プロジェクトは docs 上「企画・調査フェーズ、実装未着手」と記録されているが、
**今、実装フェーズへ移行する**。docs 内の「実装未着手」等の記述は、
あなたが実装を進めるにつれて最新状態に更新してよい（起動時にステータスを更新すること）。

# Docsの構成と扱い

事業要件・仕様・制約は `docs/` 配下に整備済み。構成は以下の3層。
- `docs/biz/` … 事業戦略・分析（concept, market, competitors, business-model, industry, usecases, roadmap, references）
- `docs/tech/` … 技術・プロダクト（tech-notes, data-model, および予約領域 customer/ business/）
- `docs/devlog/` … 作業ログ（overview.md=月次索引、YYYY-MM.md=月次ログ）

**扱いのルール：**
- `docs/biz/` の資料と `docs/overview.md` の3層構造は**意図をもって作られている。尊重し、勝手に再編・改変しない**。
- あなたが書き起こすのは主に `docs/tech/` 配下。特に **`docs/tech/customer/`（消費者向けデータ収集アプリのプロダクト/技術設計）は予約領域として用意されており、ここに実装の設計を書く**。
- 既存の biz 資料に事実誤りや実装との矛盾を見つけた場合は、勝手に書き換えず、指摘と提案を devlog に残す。

# プラットフォームと技術選定

- **プラットフォームは iOS アプリで固定**とする。ここは変更しない。
- 本アプリはまず **供給側（一般消費者向けデータ収集アプリ／`docs/tech/customer/` に対応）** を対象とする。
- iOS 以外の技術選定（言語・フレームワーク・アーキテクチャ・ライブラリ・
  テスト/静的解析ツール等）は、Docsの要件に基づき **あなたが自律的に選定してよい**。
- 選定した技術と、その選定理由・却下した選択肢を devlog に残す。

# 参考モックとデータ構造の優先順位

- `docs/reference/datamine_prototype.jsx` にモックがある（無ければ探索し、無ければその旨を記録）。
- 情報源の優先順位は厳格に次のとおり：
  **① 事業要件（docs/biz/・usecases.md）> ② data-model.md（暫定・未確定）> ③ モック（参考程度）**
- モックは使えるアイデア（UI構成・画面遷移・データの見せ方）を取り入れてよいが、
  忠実再現の義務はない。iOSアプリとして自然な形に翻案してよい。
- `docs/tech/data-model.md` は「暫定エンティティ（未確定）」である。実装に合わせて確定・更新してよいが、
  **確定させた判断はその根拠とともに devlog に残す**こと（後から覆れるように）。

# 実装の進め方（このループを回す）

**探索 → 計画 → 実装 → 検証 → レビュー → 記録 → 次のタスク**

- 大きなタスクは着手前に小タスクへ分解し、計画をログに書く。
- 実装後は必ず検証する。失敗したら直してから次へ進む。
- 1つのまとまりが完了するごとに、実装内容・判断・検証結果をログに残す。
- **一つのタスクが完了したら、確認を求めずに次のタスクへ自動で進む。**

# 検証（各まとまりの完了前に必ず通す）

- 選定した技術スタックにおける **ビルド・テスト・静的解析** をすべて通すこと。
  失敗したら直してから次へ進む。
- 使用する具体的な検証コマンドは、技術選定の結果として確定させ、devlog に明記すること。

# サブエージェントの活用方針

実装の主導権はメインエージェント（あなた）が持つ。以下はサブエージェントへ隔離する。
- 重い調査（リポジトリ探索、ドキュメント調査、ログ/テスト出力の精査）
  → 中間出力はサブエージェント側に留め、要約だけを受け取る。
- 独立した調査は、依存関係がないものに限り並列で走らせる（順序依存があるものは並列化しない）。
- 第三者レビューは reviewer サブエージェントに委ねる（下記レビューゲート参照）。
固定エージェントを増やしすぎない。隔離したい明確な理由があるものだけに絞る。

# モデルの使い分け

作業の性質に応じてモデルを使い分けること。
- 最難所の設計判断・アーキテクチャ・要件解釈・レビュー：最上位モデル（Opus系）
- 主力の実装作業：高速な上位モデル（Sonnet系）
- 機械的・定型的な作業（一括置換、命名整理、雛形生成）：軽量モデル（Haiku系）
どのモデル/エージェントに何を割り当てたかも、その理由とともに devlog に残す。

# 自律的判断のルール

実装判断は、ビジネス要件・Docs・これまでの会話から適切に行い、自律的に実装する。

**自分で決めて進めてよいこと（＝原則すべて自走）**
- 命名、内部設計、ファイル配置、リファクタリング、実装の具体的手段。
- iOS以外の技術選定。
- data-model 等、実装の土台になる未確定事項の確定（※根拠を devlog に残すこと）。
- `docs/tech/` 配下（特に customer/）のドキュメント作成・更新。

**止まって確認すべきこと（これ以外は止まらない）**
- 破壊的で復旧困難な操作（データ削除、外部への公開・送信、認証情報の扱いなど）。
- 事業の根幹（差別化3本柱＝日本特化／判断・暗黙知／閲覧権の反復販売）を
  否定する方向へ実装が進んでしまう場合。

上記2つ以外では確認を求めず、自律的に判断して進む。
判断に迷った箇所・土台を確定させた箇所は、必ず根拠を devlog に残し、後から検証可能にすること。

# レビューゲート（第三者レビュー）

実装のまとまりが完了に達したら、reviewer サブエージェントを呼び出しレビューを受ける。
- reviewer は実装過程に関与しない独立した立場で、成果物だけを白紙から評価する。
- reviewer には読み取りと検証のみを許可し、コードは書き換えさせない（指摘に徹する）。
- 🔴（Blocker）が残っている間は「完了」と呼ばない。指摘を潰してから次へ進む。
- レビュー指摘と、それへの対応（修正 or 却下の理由）も devlog に残す。

# ログの運用

- ログは **`docs/devlog/` の既存運用に従う**（月次 `YYYY-MM.md` に追記、`overview.md` の索引を更新）。
  書式・記録先の正本は docs 側にあるため、ここでは重複して定義しない。
- 実装ログに限らず、思考フロー・検討した選択肢・却下した案・判断根拠も、細かくすべて残す。

# 中断耐性（いつ切れても再開できるように）

利用制限などでセッションがいつ中断されても続きから再開できるよう、以下を徹底する。
- 各タスクの完了時点で、devlog の末尾に必ず「次にやること（Next）」を1〜3行で書く。
- 作業を「再開しやすい区切り」で止められるよう、タスクは小さく保つ。
  大きな変更の途中（ビルドが通らない中間状態）で長く留まらない。
- **再開時のプロトコル**：まず devlog の最新月次ファイル末尾を読み、
  「直前に何をどこまでやったか」「次にやること（Next）」を把握してから続行する。
- コミット可能な区切りに達したら、その単位で作業を確定させておく
  （どこまで完了したかが後から辿れる状態を保つ）。

# Git コミットの運用

- **適切な粒度でこまめに commit する**。1つの意味のあるまとまり
  （1機能・1修正・1リファクタ）を1コミットの目安とする。
  巨大な変更を1コミットに詰め込まない。ビルドが通らない中間状態でのコミットは避ける。
- **コミットメッセージは日本語で書く**。「何を・なぜ変えたか」が伝わる粒度にする。
  1行目に要約、必要なら空行を挟んで本文で理由・背景を補足する。
  例：
    `ミッション一覧画面を追加`
    `撮影UIのポカヨケ機構を実装（撮り方の失敗を構造的に防ぐため）`
- reviewer のレビューを通過し検証が済んだまとまりは、その単位で commit する。
- commit したことと、その区切りで何が完了したかは devlog にも残す
  （Git履歴と devlog の Next が食い違わないようにする）。

# 完了条件（Definition of Done）

タスクを「完了」と呼ぶ前に、以下を満たすこと。
- 対象機能が要件どおり動作する。
- 検証（ビルド・テスト・静的解析）がすべて通る。
- reviewer のレビューを受け、🔴（Blocker）が残っていない。
- 関連する docs（主に tech/）が実装と一致し、devlog に判断と結果が残っている。
