# C側アプリ 要件定義（供給側＝データ収集アプリ）- gavel

> **要約**: 供給側＝一般消費者が手元動作を撮影・投稿する**データ収集アプリ**の要件定義（正本）。決定した仕様・**なぜそう決めたか**・未確定事項をセットで記録する。`thanks_log` の `features.md` に相当。
>
> 前提: 事業モデルは [`../../biz/business-model.md`](../../biz/business-model.md)、使われ方は [`../../biz/usecases.md`](../../biz/usecases.md)、技術的裏付けは [`../tech-notes.md`](../tech-notes.md)、データ構造は [`../data-model.md`](../data-model.md)。需要側（買い手向け）は [`../business/`](../business/) を参照。

## スコープと前提（v1）

v1 は「作り込む前に、**売れるデータの型を確立して買い手に持ち込む**」ための最小構成に振り切る。

| 項目 | 決定 |
|------|------|
| **v1の狙い** | **高品質サンプル100件を作り、買い手（研究者・国内ロボット企業）に持ち込む**（[`../../biz/roadmap.md`](../../biz/roadmap.md) 最優先アクション） |
| **最初のユーザー** | **限定協力者**（招待したモニター・知人）。一般公開はしない |
| **取得データ** | **映像＋IMU（加速度・ジャイロ）**。音声アノテーションは**任意** |
| **撮影セットアップ** | **一人称・マウント式**（スマホを頭/胸のクリップに装着） |
| **データの最小単位** | **1投稿＝1タスクの1実演**（下記「中核設計A」） |
| **対象プラットフォーム** | 【要確認・**iOS優先を提案**】理由は §1 |

### v1では「やらないこと」（限定協力者前提で後回し）

一般公開を見据えた機能は、最初のユーザーが限定協力者であることを利用して意図的に外す。将来 [`../../biz/roadmap.md`](../../biz/roadmap.md) で扱う。

- レベニューシェアの本実装（ウォレット・出金・KYC・確定申告対応）
- ミッションのマーケットプレイス（多数ミッションの一覧・検索・応募）
- 多人数のコミュニティ・フレンド・ランキング
- 不正投稿の自動検知、本格的なモデレーション
- 本格的な認証基盤（パスキー/ソーシャル等）

## 中核設計A：キャプチャの最小単位（不変ルール）

**1投稿 ＝ ミッションが定義する1タスクの1実演（1デモンストレーション）。** `thanks_log` の「1投稿＝1カード」に相当する、崩さない不変ルール。

- **1投稿の構成**:

  | スロット | 中身 | 誰が埋めるか |
  |----------|------|--------------|
  | ミッション参照 | どのタスクの実演か | ミッションから自動 |
  | 映像（**必須**） | 一人称視点の手元動作クリップ（1実演） | ユーザー |
  | IMU（**必須**） | 映像と**時刻同期**した加速度・ジャイロ（＋可能なら端末姿勢） | 自動記録 |
  | 音声アノテーション（**任意**） | 「なぜそうしたか」の実況／後付けナレーション | ユーザー（任意・高単価化） |
  | メタ情報 | 撮影日時・端末・マウント種別・自己申告の成否 等 | 自動＋一部ユーザー |

- **不変ルール**:
  1. **1投稿＝1タスクの1実演**（複数タスクを1クリップに詰めない）。→ 加工・納品が均質になり、データセットとして扱いやすい。
  2. **映像とIMUは常にセットで時刻同期**（IMU単体・映像単体の投稿を作らない）。
  3. クリップは**上限尺で自動トリミング**（長尺の垂れ流しを防ぐ）。上限値は §1 未確定。
- **背景・理由**: 買い手が学習に使うのは「1タスクのきれいなデモの集合」。ここを構造で担保しないと、加工コストが跳ね、100件の均質性が崩れる。

## 中核設計B：一人称・マウント撮影（IMUを活かす前提）

- **決定**: スマホを**頭または胸の簡易クリップマウントに装着**し、**一人称視点**で撮影する。
- **背景・理由**: **IMU が意味を持つのは、カメラ（スマホ）が体と一緒に動くとき**。固定撮影ではスマホが動かず IMU がほぼ無意味になり「映像のみ」と実質同じになる（visual-inertial SLAM はカメラ軌跡推定＝カメラが動く前提。[`../tech-notes.md`](../tech-notes.md)）。参考にした **Asimov AI がヘッドバンドを採る**のも同じ理由（[`../../biz/competitors.md`](../../biz/competitors.md)）。
- **マウントの配布**: 最初のユーザーが**限定協力者**なので、簡易クリップマウントの**現物配布が現実的**。両手が空き、視点が安定し、データ品質が上がる。
- **検討中・未確定**: マウントの種別（頭 vs 胸）、両手タスクでの視野の入り方、装着ガイドの具体。

## 仕様および設計判断一覧

### 1. 対象プラットフォーム
- **提案（要確認）**: **iOS ネイティブ優先**。
- **背景・理由**: (1) 限定協力者への配布なら端末を iPhone に揃えやすく、**IMU/カメラの個体差を抑えられる**（データ品質が安定）。(2) **ARKit** が visual-inertial odometry（映像＋IMUによるカメラ姿勢推定）を標準提供し、[`../tech-notes.md`](../tech-notes.md) の狙いと直結。Android も同種（ARCore）はあるが、v1は絞る。
- **未確定**: iOS単独か、早期に Android も要るか。ネイティブかクロスプラットフォーム（センサー精度・同期の観点でネイティブ寄りを想定）。

### 2. 撮影・キャプチャUI（ポカヨケ）
- **決定**: 「撮り方の失敗を構造的に防ぐ」ポカヨケUIを中核に置く。撮影前・撮影中・撮影後の3段でガードする。
  - 撮影前チェック: マウント装着の確認、明るさ、手元が画角に入るか、IMUキャリブレーション/静止基準。
  - 撮影中ガイド: 画角外れ・急なブレ・暗さの警告、経過時間表示。
  - 撮影後の自己確認: プレビューと「今の実演は成功/失敗」の自己申告、**撮り直し**導線。
- **背景・理由**: v1の価値は「**100件が均質に高品質**」であること。品質は撮影時点で決まるため、事後加工でなく**入口で担保**する。
- **未確定**: クリップの**上限尺**（提案: 数十秒〜1〜2分程度）、IMUの**サンプリングレート**と映像との同期方式、キャリブレーション手順、失敗データ（あえて残す価値。[`../tech-notes.md`](../tech-notes.md) の失敗文脈）の扱い。

### 3. ミッション（v1は少数の常設ミッション）
- **決定**: v1 は**少数の常設ミッション**（日本特化タスク：箸使い・和食器の扱い・狭い台所での配膳 等）をアプリ内に固定表示する。各ミッションは「タスク説明・お手本・成功条件・撮り方の注意」を持つ。
- **背景・理由**: 100件づくりに集中するため、受注/常設の**マーケットプレイス的な仕組みは不要**。少数の狙ったミッションを協力者に回す方が、均質な在庫を早く積める。
- **未確定**: 初期に載せる具体的なミッションの選定（買い手ニーズと日本特化の交点。[`../../biz/usecases.md`](../../biz/usecases.md)）。

### 4. 音声アノテーション（任意）
- **決定**: 「なぜその判断をしたか」を**音声で付与**できる。v1では**任意**（付けると高単価、という事業設計の入口を体験として置く。[`../../biz/business-model.md`](../../biz/business-model.md)）。
- **未確定**: **実況型**（撮影中に喋る）か**後付け型**（撮影後にプレビューを見ながらナレーション）か、両対応か。実況は音と動作が自然に同期するが撮影負荷が上がる。

### 5. アップロード・データ管理
- **決定**: 撮影データ（映像＋IMU＋メタ）は**サーバーへアップロードし、原本を保持**する（[`../data-model.md`](../data-model.md) の Media 原本保持）。オフラインで撮影し、後でまとめて同期できる。
- **背景・理由**: 一人称の生活シーンで撮るため、電波環境に依存させない。原本保持は再加工・別アノテーションの前提。
- **未確定**: 端末側の一時保存容量・圧縮方針、アップロードの単位（実演ごと/まとめて）。

### 6. 報酬（v1は最小）
- **決定**: v1は**レベニューシェアの本実装を行わない**。限定協力者への**固定謝礼＋貢献記録（何件・どのミッションを提供したか）**に留める。
- **背景・理由**: 「売れるたびに還元」はデータが実際に売れて初めて回る仕組み。v1のゴールは「売れる型づくり」であり、売買はまだ発生しない。ウォレット・出金・KYCは公開フェーズの課題（[`../../biz/roadmap.md`](../../biz/roadmap.md)）。
- **未確定**: 固定謝礼の水準、貢献記録をどこまで見せるか。

### 7. 安全・プライバシー
- **決定**: 限定協力者前提でも、**同意と写り込み配慮**は最初から置く。
  - 協力者との**データ利用・再ライセンスの同意**（撮ったデータを企業へ閲覧許諾する旨）。
  - 一人称視点ゆえ**自分の顔は入りにくい**が、**家族・第三者・自宅・住所特定要素**が写り込む。撮影ガイドで最小化し、**投稿前の人力確認**を挟む。
- **背景・理由**: 実務上の法的・契約的設計は事業側の宿題（[`../../biz/roadmap.md`](../../biz/roadmap.md) §4）。v1は「同意書＋撮影ガイド＋人力確認」で軽く担保し、端末側の自動顔ぼかし等は公開フェーズで実装。
- **未確定**: 同意フローの具体、第三者写り込みの許容線、未成年協力者を含めるか。

### 8. アカウント・オンボーディング
- **決定**:
  - v1のアカウントは**招待コード＋最小限の識別**に留める（本格的な認証基盤は後回し）。
  - **オンボーディング（良い撮影の教示）を重視する**：マウント装着、視点、明るさ、1実演の撮り方、成功/失敗の自己判断。
- **背景・理由**: 品質が命であり、かつ協力者は撮影に不慣れ。**最初の数件で撮り方を体に入れてもらう**ことが100件の均質性を左右する。認証は限定配布なので軽くてよい。
- **未確定**: 識別方式（招待コードのみ/簡易ログイン）、オンボーディングを何件目まで手厚くするか。
